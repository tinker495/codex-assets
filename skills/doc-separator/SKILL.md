# Document Separator: Tobe vs As-Is 분리

## 목적

하나의 문서에 섞여있는 **Tobe 스펙**(설계 의도, 목표 상태)과 **As-Is 기술**(현재 코드가 실제로 하는 것)을 두 개의 독립된 문서로 분리한다.

## 왜 분리해야 하는가

섞인 문서는 다음 문제를 일으킨다:
- 에이전트가 코드를 작성할 때 "현재 상태"와 "목표 상태"를 혼동
- Drift를 발견할 수 없음 (비교 대상이 분리되지 않았으므로)
- 리뷰어가 "이게 의도인지 현재 동작인지" 판단 불가

## 분류 기준

### Tobe (설계 의도/목표) 로 분류할 시그널:
- "~해야 한다", "~할 것이다", "~하도록 설계"
- "목표", "의도", "계획", "설계", "스펙", "요구사항"
- 조건부 미래형: "구현되면 ~", "최종적으로는 ~"
- 구체적 코드 참조 없이 개념적/추상적으로 기술된 동작 설명
- KPI 목표치, 성능 목표, 제약조건 정의

### As-Is (현재 실제 동작) 로 분류할 시그널:
- "현재", "지금은", "실제로는", "구현 상태"
- 구체적 함수명, 클래스명, 변수명 언급
- 코드 스니펫, 실행 결과, 로그 출력
- "~하고 있다" (현재 진행형으로 코드 동작 설명)
- 버그 리포트, 알려진 이슈, workaround 설명
- 실측 성능 수치 (예: "2502E에서 hard violation 0")

### 애매한 경우 처리:
- **판단 불가**: `[AMBIGUOUS]` 태그를 붙이고 양쪽 문서 모두에 포함. 사람이 판단하도록 남김.
- **양쪽 모두 해당**: 양쪽에 각각 포함하되, 상호 참조 표시.

## 출력 형식

입력 문서 하나에 대해 두 개의 마크다운 파일을 생성한다:

### 파일 1: `{module_name}_TOBE.md`

```markdown
# {모듈명} - Tobe 스펙

## 개요
(이 모듈이 달성해야 할 목표)

## 설계 의도
(원본 문서에서 추출한 의도/목표 기술)

## 목표 입출력
(정의된 경우)

## 목표 제약조건
(정의된 경우)

## 미구현 항목
(문서에 언급되었으나 As-Is에서 확인 안 되는 것)
```

### 파일 2: `{module_name}_ASIS.md`

```markdown
# {모듈명} - As-Is 현재 상태

## 개요
(이 모듈이 현재 실제로 하는 것)

## 현재 동작
(원본 문서에서 추출한 현재 상태 기술)

## 현재 입출력
(확인된 경우)

## 알려진 이슈 / Workaround
(문서에 언급된 경우)

## [AMBIGUOUS] 미분류 항목
(판단 불가 항목 모음)
```

## 워크플로우

1. 대상 문서를 전체 읽는다.
2. 문단/섹션 단위로 위 분류 기준에 따라 태깅한다.
3. 태깅 결과를 두 파일로 분리 출력한다.
4. `[AMBIGUOUS]` 항목이 있으면 목록을 사용자에게 제시하고 판단을 요청한다.
5. 원본 문서의 어떤 내용도 누락하지 않는다. 모든 내용은 Tobe, As-Is, 또는 AMBIGUOUS 중 하나에 반드시 포함.

## 주의사항

- 원본 문서의 의미를 해석하거나 보강하지 않는다. 있는 그대로 분류만 한다.
- 코드에서 추가 정보를 읽어오지 않는다. 이 스킬은 문서만 대상으로 한다.
- As-Is 문서가 코드와 일치하는지 검증은 이 스킬의 범위가 아니다 (-> reverse-doc 스킬에서 처리).
